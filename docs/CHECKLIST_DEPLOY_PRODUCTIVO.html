Crítico 1. Decisiones del Equipo (Prerequisitos)

Estas decisiones deben tomarse antes de iniciar el trabajo técnico. Sin ellas no se puede avanzar.
  • Definir dominio de producción
    ¿Tienen dominio propio? (ej: plataforma.mso.cl, tpt.msochile.cl). Si no, definir si se compra uno o se usa subdominio. El dominio es necesario para configurar SSL, correos y la URL pública de la plataforma.
    Equipo MSO
  • Elegir plataforma de hosting
    Opciones disponibles:
    OpciónCosto aprox.VentajasDesventajas
    Vercel (recomendado)Gratis / $20 USD/mes ProDeploy automático desde GitHub, SSL gratis, Edge Functions, excelente performanceRequiere cuenta
    NetlifyGratis / $19 USD/mes ProSimilar a Vercel, buen panel de controlEdge Functions más limitadas
    GitHub PagesGratisYa lo usan, cero configuraciónSolo estático, sin server-side, sin Edge Functions
    Servidor propio (VPS)$5-20 USD/mesControl totalRequiere mantención, SSL manual, más trabajo
    Equipo MSO
  • Crear proyecto Supabase de producción (separado del demo)
    Ir a supabase.com → New Project. Región recomendada: South America (São Paulo) para menor latencia desde Chile. Guardar URL del proyecto y ambas API keys (anon + service_role). El proyecto demo actual NO debe usarse en producción.
    Equipo MSO
  • Definir plan de Supabase
    PlanCostoLímites clave
    Free$0500MB BD, 1GB storage, 50K auth users, pausado tras 1 semana inactividad
    Pro (recomendado)$25 USD/mes8GB BD, 100GB storage, sin pausa, backups diarios, soporte email
    Team$599 USD/mesPara equipos grandes, SOC2, soporte prioritario
    Para producción real el Plan Pro es el mínimo recomendado (el plan Free pausa el proyecto tras inactividad).
    Equipo MSO
  • Configurar servicio SMTP para correos transaccionales
    Necesario para: confirmación de cuenta, recuperación de contraseña, notificaciones por email. Opciones: Resend (gratis hasta 3K/mes), SendGrid (gratis hasta 100/día), Mailgun, o SMTP propio de la organización. Configurar en Supabase → Settings → Auth → SMTP.
    Equipo MSO
  • Obtener API key de Groq para producción
    La key actual está expuesta en código público y puede ser revocada por Groq. Crear una nueva key exclusiva para producción en console.groq.com. Esta key NO irá en el frontend, sino en una Edge Function server-side. Costo: Groq es gratuito actualmente con límites de rate.
    Equipo MSO
  • Preparar listado de usuarios iniciales
    Excel o CSV con columnas: nombre, email, rol (admin/jefatura/participante), cargo, cliente. Estos serán los primeros usuarios con cuentas reales en la plataforma. Definir también quién será administrador.
    Equipo MSO
  • Definir política de contraseñas
    Largo mínimo (recomendado: 8 caracteres), requisitos de complejidad, ¿permitir recuperación por email?, ¿forzar cambio en primer login?
    Equipo MSO

Crítico 2. Seguridad (Implementación Técnica)

Cambios obligatorios antes de exponer la plataforma a usuarios reales. Estas tareas las implementa Claude/desarrollador.
  • Migrar autenticación a Supabase Auth
    Reemplazar login contra columna password_visible (texto plano) por supabase.auth.signInWithPassword(). Las contraseñas se almacenarán con hash bcrypt en auth.users. Eliminar columna password_visible de la tabla usuarios.
    Claude
  • Configurar Row Level Security (RLS) por rol
    Definir políticas de acceso:
    • Admin: lectura/escritura en todas las tablas
    • Jefatura: lectura de sus programas, sus participantes, escritura de feedback y coevaluaciones
    • Participante: lectura de su perfil/encuestas/actividades, escritura de sus respuestas
    • Anon: solo acceso a login/registro
    Vincular políticas con auth.uid() y el campo auth_id de la tabla usuarios.
    Claude
  • Mover Groq API key a Edge Function
    Crear Supabase Edge Function (Deno) que reciba el prompt del frontend y haga la llamada a Groq server-side. La API key se almacena como secret en Supabase, nunca en el código del browser. El frontend llama a /functions/v1/generar-informe en vez de directamente a api.groq.com.
    Claude
  • Revocar API keys expuestas
    La Groq API key debe vivir exclusivamente como secret de Supabase (GROQ_API_KEY) y consumirse via Edge Function proxy. Nunca en el frontend.
    Equipo MSO
  • Configurar CORS y dominios permitidos
    En Supabase → Settings → API, agregar solo el dominio de producción como origen permitido. Esto previene que otros sitios usen la API.
    Claude
IMPORTANTE: Mientras la plataforma use contraseñas en texto plano y RLS deshabilitado, cualquier persona con conocimientos básicos de browser DevTools puede ver TODAS las contraseñas y datos de TODOS los usuarios. No exponer a usuarios reales hasta completar esta sección.

Importante 3. Base de Datos Productiva

Configuración de la base de datos para producción.
  • Ejecutar schema en proyecto Supabase de producción
    Ejecutar supabase-schema-v2.sql (sin datos demo) en el SQL Editor del proyecto productivo. Verificar que las 10 tablas se creen correctamente.
    Claude
  • Ejecutar script de RLS productivo
    Aplicar las políticas de seguridad por rol después de crear las tablas. Script generado por Claude.
    Claude
  • Cargar datos iniciales
    Insertar clientes, usuarios iniciales (con Supabase Auth), y programas base. Usar el listado de usuarios preparado por el equipo.
    Ambos
  • Configurar backups automáticos
    En Plan Pro, los backups diarios están incluidos. Verificar que estén activos en Supabase → Settings → Database → Backups. Considerar también export manual periódico (pg_dump) para respaldo adicional.
    Equipo MSO
  • Agregar tablas de auditoría (opcional)
    Tabla audit_log para registrar quién hizo qué y cuándo: creación/edición/eliminación de registros, cambios de estado de encuestas, etc.
    Claude

Importante 4. Frontend y Dependencias

Optimizaciones para que el frontend sea estable y profesional en producción.
  • Reemplazar Tailwind CDN Play por versión estable
    El CDN Play de Tailwind no está recomendado para producción (puede fallar, es lento). Opciones: compilar Tailwind con PostCSS (build step) o usar el CDN estándar con clases pre-compiladas. Recomendado: usar cdn.tailwindcss.com versión pinneada.
    Claude
  • Pinnear versiones de dependencias CDN
    Fijar versiones específicas con SRI hashes para todas las librerías:
    • Chart.js (actualmente: latest)
    • Supabase JS SDK v2 (actualmente: latest)
    • SheetJS (ya pinneado: 0.18.5)
    • html2pdf.js (ya pinneado: 0.10.1)
    Esto previene que una actualización de la librería rompa la plataforma.
    Claude
  • Eliminar archivo mock.js del deploy productivo
    El archivo mock.js contiene datos demo hardcodeados y la API key de Groq. No debe estar en producción. Solo mantener supabase-client.js.
    Claude
  • Actualizar supabase-client.js con credenciales de producción
    Reemplazar URL y anon key del proyecto demo por las del proyecto productivo. Idealmente usar variables de entorno si el hosting lo permite.
    Claude
  • Eliminar archivos .sql del deploy
    Los archivos supabase-schema.sql y supabase-schema-v2.sql no deben estar accesibles públicamente ya que revelan la estructura de la BD.
    Claude
  • Agregar página de error y manejo offline
    Página 404 personalizada y mensaje amigable cuando no hay conexión a Supabase.
    Claude

Recomendado 6. Hosting, Dominio y DNS

Configuración del entorno productivo.
  • Registrar/configurar dominio
    Si ya tienen dominio (ej: msochile.cl), crear subdominio plataforma.msochile.cl. Si no, registrar uno nuevo (~$10-15 USD/año en NIC Chile para .cl).
    Equipo MSO
  • Configurar hosting y deploy automático
    Conectar repositorio GitHub al hosting elegido. Cada push a main despliega automáticamente. Configurar variables de entorno (SUPABASE_URL, SUPABASE_KEY).
    Claude
  • Configurar SSL / HTTPS
    Obligatorio para producción. Vercel y Netlify lo incluyen gratis automáticamente. GitHub Pages también con dominio personalizado.
    Claude

Recomendado 7. Testing y QA

Verificación antes del lanzamiento.
  • Test de flujo completo: Admin
    Login → Crear cliente → Crear programa → Agregar participantes → Crear competencias → Crear encuesta → Activar encuesta → Ver dashboard → Generar informe IA
    Ambos
  • Test de flujo completo: Jefatura
    Login → Ver inicio → Responder coevaluación → Ver equipo → Dar feedback → Descargar informe
    Ambos
  • Test de flujo completo: Participante
    Login → Ver encuestas pendientes → Responder encuesta PRE → Ver progreso → Ver feedback recibido → Responder encuesta POST
    Ambos
  • Test de seguridad básico
    Verificar que un participante NO puede ver datos de otros participantes. Verificar que jefatura NO puede acceder a funciones de admin. Verificar que no hay API keys en el código fuente del browser.
    Claude
  • Test en móviles y tablets
    Verificar que la plataforma es usable en dispositivos móviles (encuestas se responden desde celular frecuentemente). Probar en Chrome, Safari, Firefox.
    Ambos

Lanzamiento 8. Go-Live y Post-Lanzamiento

Pasos finales para salir a producción.
  • Carga masiva de usuarios reales
    Importar usuarios desde Excel con creación automática de cuentas en Supabase Auth. Enviar email de bienvenida con credenciales temporales.
    Ambos
  • Crear programas y encuestas reales
    Configurar el primer programa productivo con competencias, encuestas PRE/POST y participantes asignados.
    Equipo MSO
  • Monitoreo post-lanzamiento
    Revisar logs de Supabase las primeras 48 horas. Verificar que las encuestas se guardan correctamente. Estar atento a errores reportados por usuarios.
    Ambos
  • Documentación de uso
    Guía rápida para administradores: cómo crear programas, cómo agregar participantes, cómo activar encuestas, cómo generar informes.
    Ambos
